# 交割智慧监管平台流程重构建议

## 1. 背景

本建议用于重构当前 `交割智慧监管平台.json` 的业务建模口径，重点解决以下问题：

- 流程切分过粗，不符合流程引擎/低代码平台的流程模板视角。
- 多个“管理类流程”把新增、修改、禁用、查询混在一个流程里，导致一个流程实例承担了多个业务目标。
- 仓储仓单主线缺少“仓单注册 / 仓单注销 / 仓单过户 / 仓单链查询”等核心流程。
- 现货仓单、期货仓单、厂库仓单、期现关联之间的对象边界不够清晰。

本轮梳理基于以下资料：

- 当前工作区文档：`workspace/交割智慧监管平台.json`
- 业务分析资料：`C:\Users\Administrator\Desktop\智慧交割\设计文档\docs\探讨：交割业务的分析设计.docx`
- 范围交叉校验：`C:\Users\Administrator\Desktop\智慧交割\设计文档\docs\交割智慧监管平台.pdf`

## 2. 核心判断

### 2.1 流程模板的切分原则

从 Flowable 一类流程引擎的视角，一个流程模板应满足：

- 一个流程实例只解决一个明确业务目标。
- 流程结束后，业务结果是单一、清晰、可判断的。
- 流程内部允许有多个节点，但不能混入另一个独立业务目标。

因此，下面这种建模方式不再合适：

- `仓库主体信息管理`
- `仓房垛位与提货地点管理`
- `基础信息管理`

因为这些流程实质上混合了：

- 新增
- 修改
- 启停/禁用
- 查询

它们应该被拆成多个流程模板，并用 `流程组` 包起来。

### 2.2 查询流程不能缺位

查询不只是“页面功能”，在监管平台里它同样是一个稳定业务动作：

- 有明确的查询入口和权限边界
- 有稳定的输入条件
- 有明确的结果视图
- 常常承载审计、监管、比对、追溯价值

所以查询流程应该保留，不应因为“不是办理类流程”就忽略。

### 2.3 仓储仓单的真正主线不是“管理”，而是“领域事件”

从《探讨：交割业务的分析设计》来看，仓单交割主线不是一个笼统的“仓储仓单管理”，而是围绕以下事件演化：

- 入库
- 货转
- 注册
- 过户
- 注销
- 出库
- 移垛
- 盘库
- 期现关联
- 仓单链查询

其中尤其要注意：

- `现货货转 / 注册 / 注销 / 出库` 会产生新的仓储仓单管理单元。
- `移垛` 只改变位置，不改变总量和货权。
- `盘库` 是审计留痕，不直接改变仓单数量。

## 3. 当前模型的主要问题

### 3.1 业务子域数量失真

当前文档里混入了示例子域 `图书馆`，导致业务子域数变成 11。

交割平台的正式建模应只保留这 10 个业务子域：

1. 用户管理
2. 基础数据管理
3. 交割服务机构管理
4. 仓储仓单管理
5. 厂库库存管理
6. 车船板交割管理
7. 电子仓单同步数据管理
8. 视频监控管理
9. 物联网设备管理
10. 综合大屏

### 3.2 交割服务机构管理切分过粗

当前存在：

- `仓库主体信息管理`
- `仓房垛位与提货地点管理`

这两类流程都不是单一目标流程，更像“流程组”。

### 3.3 仓储仓单管理主链缺项

当前已有：

- 入库预约管理
- 入库办理
- 入库盘点与信息比对
- 在库监控与全景查询
- 期现关联管理
- 现货货转管理
- 移垛管理
- 盘库管理
- 货权人查库管理
- 出库预约管理
- 出库办理

但缺少以下核心流程：

- 仓单注册
- 仓单注销
- 仓单过户
- 仓单链查询
- 期现差异查询/核对

这会导致当前模型无法覆盖交割主线。

### 3.4 实体边界仍不够准确

当前实体中已有：

- `现货仓单`
- `期现关联`
- `厂库仓单`

但还不够，至少还缺少：

- 仓库场景下的 `期货仓单`
- 用于描述父子拆分/合并血缘的 `仓储仓单关系`
- `仓单注册记录`
- `仓单注销记录`
- `仓单过户记录`

## 4. 重构后的流程切分原则

### 4.1 一级结构

仍采用：

- 业务子域
- 流程组
- 流程

### 4.2 命名规则

- `流程组`：表达同类业务能力，例如 `仓库主体维护`
- `流程`：表达单一业务目标，例如 `新增仓库主体`

### 4.3 适合保留为单流程的类型

- 申请-审核-结果闭环类
- 同步接收-校验-落库类
- 查询分析类
- 监管事项闭环类

### 4.4 需要拆开的类型

- 一个流程里同时包含新增、修改、禁用、查询
- 一个流程里同时包含不同业务结果
- 一个流程里既做“维护”又做“监管闭环”

## 6. 仓储仓单实体重构建议

## 6.1 建议的核心对象

### 现货仓单

用于表示“货物位置 + 货物数量”的实际管理单元。

建议保留当前 `现货仓单`，但加强以下语义：

- 是仓储侧的核心实体
- 具备父子血缘
- 支持入库、货转、注册、注销、出库产生新单元
- 支持移垛但移垛不产生新数量

### 仓库期货仓单

建议新增，用于表示仓库场景下已注册到期货侧的标准仓单。

建议关键字段：

- 期货仓单ID
- SPC_ID / 非通用仓单编号
- 关联现货仓单ID
- 仓单状态
- 注册时间
- 注销时间

### 厂库期货仓单

建议将当前 `厂库仓单` 语义明确为 `厂库期货仓单`。

原因：

- 厂库场景不存在实体在库货物定位
- 但仍然是期货侧流通对象

### 期现关联

建议保留，但从“泛关联”改为“强一一映射”。

建议关键字段：

- 期现关联ID
- 现货仓单ID
- 仓库期货仓单ID
- SPC_ID
- 关联确认方式
- 关联状态
- 生效时间
- 失效时间

建议规则：

- 同一有效 `SPC_ID` 只能关联一个当前有效现货仓单
- 同一当前有效现货仓单只能关联一个当前有效 `SPC_ID`

### 仓储仓单关系

建议新增，用于表达仓单链。

建议关键字段：

- 关系ID
- 父现货仓单ID
- 子现货仓单ID
- 关系类型
- 触发动作
- 生效时间

关系类型建议至少覆盖：

- 入库形成
- 货转拆分
- 注册拆分
- 注销回流
- 出库结转

## 6.2 关键业务规则

### 规则 1：期现关联是一一对应

当前模型里 `期现关联` 只有 `期货仓单号` 字符字段，不足以表达真正的一一映射。

应提升为：

- `现货仓单ID ↔ SPC_ID`
- 一一对应
- 需要人工确认
- 需要保留有效期与失效期

### 规则 2：新仓单单元的产生

以下动作会产生新的现货仓单管理单元：

- 现货货转
- 注册
- 注销
- 出库

因此系统不能只更新“原现货仓单状态”，还需要记录父子关系。

### 规则 3：移垛只改位置

移垛不会改变：

- 数量
- 货权
- 父子血缘

移垛只改变：

- 垛位
- 仓房
- 位置留痕

### 规则 4：盘库是审计留痕

盘库记录用于：

- 核查
- 留痕
- 触发监管关注

但不应直接改写现货仓单数量。

## 7. 当前模型优先改造顺序

建议按以下顺序落地：

1. 从 `交割智慧监管平台.json` 中移除 `图书馆` 示例子域与实体
2. 先补 `flowGroup`
3. 再拆 `交割服务机构管理` 中的“管理类大流程”
4. 补齐 `仓储仓单管理` 的 `仓单注册 / 仓单注销 / 仓单过户 / 仓单链查询`

## 8. 第二轮细化建议（面向流程引擎模板粒度）

在第一轮完成 `业务子域 -> 流程组 -> 流程` 之后，仍需继续把偏粗的“管理类流程”压到更接近流程引擎模板的粒度。

### 8.1 已适合继续拆开的流程

- `账号维护` -> `新增账号 / 修改账号 / 启停账号 / 查询账号`
- `角色维护` -> `新增角色 / 修改角色 / 启停角色 / 查询角色`
- `基础信息管理` -> `新增基础信息项 / 修改基础信息项 / 启停基础信息项 / 查询基础信息项`
- `参数配置管理` -> `新增参数配置 / 修改参数配置 / 启停参数配置 / 查询参数配置`
- `数据字典管理` -> `新增字典项 / 修改字典项 / 启停字典项 / 查询字典项`
- `商品主数据管理` -> 至少拆为品牌与等级规格两组原子流程
- `入库预约变更撤销` -> `入库预约变更 / 入库预约撤销`
- `出库预约变更撤销` -> `出库预约变更 / 出库预约撤销`
- `厂库出库预约管理` -> `厂库出库预约申请 / 厂库出库预约变更 / 厂库出库预约撤销 / 厂库出库办理`
- `车船板预报与配对接入` -> `车船板预报接入 / 交割配对确认`
- `现场签到与摇号抽样` -> `现场签到 / 摇号抽样`
- `复检申请与押金处理` -> `复检申请 / 押金处理`

### 8.2 第二轮的判断标准

- 一个流程实例只做一件事
- 一个流程结束后只有一个清晰结果
- 不再把“变更 + 撤销”“接入 + 确认”“签到 + 抽样”放在同一个模板里
- 查询继续保留为独立流程，不回退到“维护大流程”的附属动作
5. 最后调整实体模型，补 `仓库期货仓单` 与 `仓储仓单关系`
