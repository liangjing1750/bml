# 业务建模思考

我的理解是：业务建模是反复校正认知，螺旋式前进的过程：

- 自上而下：先沿业务时间线和价值流梳理全景，知道业务大体怎么发生、怎么创造价值。
- 自下而上：再从“流程、数据、规则”里补充细节，反过来修正前面对业务的理解。

BLM工具，吸收了《聚合架构》这本书的以下观点：
- 数据长在流程上，业务架构包含数据架构。
- 以一个角色连续的动作序列划分流程节点。
- DDD更偏向于技术人员的领域建模，业务人员更需要流程和数据的视角。

## 1. 业务按照时间线/流程梳理

建模刚开始时，不建议先陷入数据库、字段、页面和系统功能。

更自然的思考过程是不断的反问：

> 在这个时间点，什么角色，干了什么。

这句话有两个作用。第一，它帮助团队先梳理业务时间线。第二，它把角色、动作、结果先拉到台面上，避免一开始就进入系统实现细节。

以交割智慧监管业务为例，最开始看到的往往是一条粗时间线：

`货物入库 -> 货物在库 -> 货物出库`

继续往下看，入库不是一步完成的。它可能拆成入库预约、审核回复、办理入库、生成或归档单据。

- 再往前追，想要预约，必须先是平台用户，于是前面又会补出“用户管理”。
- 再往下拆，办理入库会继续拆成不同角色连续动作形成的流程节点。
- 再往旁边看，入库材料可能需要视频截图、视频回放，于是又会识别出“视频监控”这样的支撑能力。

这个例子说明，业务不是一条线就能解释完。

时间线能帮我们看先后顺序，能力边界能帮我们看哪些事情应该归到同一块，建模时要允许自己来回修正。

比如：一开始看起来像一个流程，细化以后可能发现它跨了两个角色职责，需要拆成两个节点。

## 2. 流程的五级模型定义

五级模型是：价值流、阶段、流程、流程节点、任务/步骤。

| 层级 | 主要回答的问题 | 粗略量化 |
| --- | --- | --- |
| 价值流 | 业务靠哪些大环节创造价值？ | 通常 5-7 个，能覆盖全域 |
| 阶段 | 业务走到哪一段？这一段承载什么能力包？ | 一个阶段能容纳多个流程 |
| 流程 | 一个业务事项如何被发起、流转、办结？ | 能产生流程实例 |
| 流程节点 | 某个角色负责哪一段连续动作？ | 角色、目标、上下文保持连续 |
| 任务/步骤 | 节点内部怎么操作或执行？ | 用户可点击，系统可执行 |

价值流适合放在业务全景图的横轴上。它不追求执行顺序里的每个细节，也不直接落到工作流引擎。它更像企业架构里的业务地图，用来说明价值如何从前到后流动。价值流太少，看不出业务差异；太多，又会退化成流程清单。

阶段更像价值流下的稳定业务能力包，也常常带有生命周期含义。比如 `仓库管理`、`品种参数管理`、`仓单注册`、`仓单注销`、`风险监管`，都可以作为阶段或阶段卡片。阶段可以表达先后，也可以表达分流，但这里的分流只是业务关系，不需要引入 BPMN 网关那样复杂的控制逻辑。

流程是可实例化的办理模板。它围绕一个主要业务对象，由触发事件启动，经过角色、系统或规则节点，使业务对象从一个状态进入另一个状态。流程才更接近低代码平台、Flowable 或 BPMN 模板。

流程节点是一个角色在一个责任边界内的连续动作。这个标准比较好用：换角色、换目标、等待他人、进入审批、切到系统自动处理、产生关键状态变化，通常都意味着节点边界出现了。

任务和步骤是节点内部的细节。这里我倾向于拆成两个视角：用户操作步骤面向产品和业务人员，描述查看、填写、选择、上传、确认、提交；编排任务面向研发和低代码实施，描述查询、校验、计算、服务调用、状态变更、通知、补偿。两者不一定一一对应。

## 3. 子域、阶段、流程和流程组的区别

这里最容易混的是业务子域、业务阶段、业务流程和流程组。

业务子域回答的是：

> 这类事情归哪一块业务能力负责？

它更接近 DDD 里的业务域或子域视角，强调能力边界。核心子域直接承载业务主线和差异化能力，例如交割服务机构管理、仓储仓单管理、厂库库存管理。通用子域为多个阶段提供支撑，例如用户账号管理、基础数据管理、视频设备管理、视频监控。

业务子域不是按时间先后划分的。“视频监控”可能在入库、在库、出库多个阶段都会被使用。它不是某个阶段的下级，而是一种可被多个阶段借用的能力。

业务阶段回答的是：

> 业务走到哪一步了？这一段稳定承载什么能力？

阶段更像业务全景图里的时间线和能力包。它帮助我们把一个大的业务过程切成几段，让团队先看到业务从哪里开始、经过哪些关键阶段、最后走向哪里。

在交割监管场景中，阶段可以从粗到细逐步形成。粗粒度可以是货物入库、货物在库、货物出库。中粒度可以是入库预约、审核回复、办理入库。更细的内容要看实际管理需要，不必一开始就拆到最细。

业务流程回答的是：

> 为了得到一个明确业务结果，业务通常怎么做？

一个流程通常有相对稳定的输入、处理过程、输出和业务结果。比如“入库预约”可以是一个流程，因为它有发起角色、输入材料、处理动作、校验规则和结果状态。

阶段和流程的判断，我会用一个简单标准：

> 能不能产生流程实例？能，就是流程。只是用来归类多个流程、组织业务视图、表达能力边界，大概率是阶段。

`仓单注册` 这个词就很典型。它既可以是阶段，也可以是流程名，取决于颗粒度。如果下面还有 `仓库仓单注册`、`厂库仓单注册`、`免检品牌仓单注册`，那 `仓单注册` 更适合作为阶段；下面这些能被发起和办结的事项，才是流程。

判断一个概念是不是流程，可以看几件事：

- 有没有触发事件或发起人。
- 有没有主要业务对象，比如仓单、仓库、品种、合约、交割预报。
- 有没有开始状态和结束状态。
- 有没有角色到角色、角色到系统、系统到角色的责任转移。
- 有没有待办、审批、记录、留痕或办结结果。
- 未来能不能变成低代码流程模板。

从工具设计上看，流程最好既有能力归属，也能被阶段引用。业务子域管理流程的能力归属，业务阶段管理流程在时间线中的组织和先后关系。阶段视图中的流程卡片，本质上是流程在这个阶段里的使用位置。

流程组则要轻一些。它回答的是“流程数量多了以后，怎样更方便浏览和维护”。流程组更像流程库中的主题归拢，不建议把它放到正式业务架构层级里。讨论能力边界时看子域，讨论业务先后时看阶段，只是为了收拢一批流程时才用流程组。

## 4. 表单、实体的区别

表单不是实体。这个边界很重要。

实体回答的是：

> 业务世界里有哪些稳定对象，它们有什么状态、规则和生命周期？

表单回答的是：

> 某个场景下，用户需要看什么、填什么、确认什么？

比如 `仓库` 是实体，但可能有多个表单使用它：仓库新增表单、仓库变更表单、仓库查询表单、仓库启停用表单、仓单注册时的仓库选择区。同一个实体字段，在不同表单里的表现也不同。有时必填，有时只读，有时隐藏，有时是查询条件，有时是审批展示字段。

我觉得更稳的关系是：

- 实体字段沉淀标准事实，例如 `warehouse.name`、`warehouse.capacity`。
- 表单字段表达当前场景，例如“仓库名称”，可以绑定到实体字段。
- 表单挂在流程节点上，说明这个节点里谁看、谁填、谁确认。
- 节点决定当前角色、当前上下文和提交后的业务结果。
- 编排任务负责提交后的查询、校验、写入、通知和流转。

一个表单可能写多个实体，一个实体也可能被多个表单维护。所以不要从表单直接倒推出实体，更不要把数据库表简单等同于页面表单。

这也影响产品和研发的分工。

产品经理不只是画页面，而是要把“业务人员如何办事”说清楚，包括用户步骤、表单定义、字段含义、校验口径、办理路径和异常场景。研发也不只是建表，而是要把“系统如何保证这件事成立”说清楚，包括实体、聚合、状态机、接口、事务一致性、权限、审计、幂等和补偿。

中间最好有一个架构或 BA 的翻译层。它负责把用户步骤抽成流程节点，把表单字段映射到实体字段，把状态变化沉淀到实体生命周期里，把流程规则和领域规则分开放。否则低代码平台很容易把领域模型拖成一堆页面字段。

一句话说，产品经理定义“业务人员如何办事”，研发定义“系统如何保证这件事可靠成立”，架构师负责让流程、表单、实体三套模型对齐。

## 5. 使用BLM工具

BLM工具里，不试图用一张图解决所有问题，而是保留几种互相引用的模型。

- 全景视图承载价值流和阶段，让团队先看到业务地图。
- 子域视图帮助团队看清业务能力边界。
- 流程模型承载可实例化的办理事项，后续可以靠近 Flowable 或低代码流程模板。
- 节点模型承载角色职责，是业务人员和研发人员对话的最小责任单元。
- 表单模型挂在节点下，表达产品和业务视角的输入输出。
- 编排任务挂在节点下，表达研发和低代码实施视角的系统动作。
- 实体模型独立维护，再被表单、节点和编排任务引用。

这样做的好处是，模型之间有关联，但不会互相吞掉。流程图不是页面原型，表单模型不是领域模型，实体模型也不能替代业务人员对办理过程的理解。

团队使用时不用把所有概念都做全，可以先用简单结构降低使用成本，随着模型复杂度上升，再逐步增强阶段引用、跨阶段复用、全景视图和流程详情图。
让团队在建模时能不断把认知沉淀下来，并且能在后续讨论中继续修正。

实际建模时，可以按这个顺序推进：

1. 先写出业务的一条粗时间线或价值流。
2. 把价值流切成几个业务阶段。
3. 找出每个阶段里最关键、可实例化的业务流程。
4. 回头整理业务子域，确认这些流程归属于哪些能力边界。
5. 对重点流程继续拆节点，明确角色职责。
6. 对节点继续拆用户步骤、表单、编排任务、数据、状态和规则。
7. 把表单字段映射到实体字段，把流程状态变化映射到实体生命周期。
8. 最后再回头校正价值流、阶段、子域和流程颗粒度。

这套顺序不是硬性流程，只是一个帮助团队开始讨论的抓手。

真正重要的是：每一轮建模之后，团队都应该比上一轮更清楚业务边界、业务顺序、角色职责、表单语义、实体事实和规则落点。
