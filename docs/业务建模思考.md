# 业务建模思考

业务建模，不是一次性把图画完，而是一个自上而下和自下而上反复校正的过程。  
自上而下，是先沿业务时间线梳理全景；自下而上，是在流程、节点、步骤、数据和规则不断细化后，再回头修正前面对业务的理解。

一开始，团队通常不是先从字段和页面入手，而是先问一句很朴素的话：`什么角色，干了什么。`  
这句话有两个作用。第一，它帮助团队先顺出业务时间线。第二，它能把“角色、动作、结果”先拉到台面上，避免一开始就陷入系统细节。

以交割监管为例，最开始看到的往往是一条粗时间线：`货物入库 -> 货物在库 -> 货物出库`。  
继续往下想，入库不是一步完成，还会拆出预约、审核回复、办理入库等更细的阶段。再往前追，想要预约，必须先是平台用户，于是前面又会补出用户管理。再往下拆，办理入库会继续拆成不同角色连续动作的节点。再往旁边看，入库材料需要视频截图等内容，于是又会识别出视频监控这样的支撑能力。

在这个过程中，`业务阶段` 和 `业务子域` 是两个不同的管理视角。  
`业务阶段` 解决的是时间线问题，回答“业务走到哪一步了，前后顺序是什么”。  
`业务子域` 解决的是能力边界问题，回答“这类事情归哪一块业务能力负责”。  
二者不是互斥关系，而是交叉存在的。一个阶段会借助多个子域的能力，一个子域也可能服务多个阶段。

`业务子域` 更偏业务能力划分。  
它强调边界，尽量不重叠。通常可以分成两类：一类是偏业务主线的核心子域，例如交割服务机构管理；另一类是偏支撑性的通用子域，例如用户账号管理、视频设备管理、视频监控等。

`业务流程` 是连接时间线和能力边界的核心对象。  
它不是单纯的时间节点，也不是单纯的能力名称，而是为了得到一个明确业务结果而形成的一种相对稳定的做法。流程通常有比较清晰的输入、输出和结果，因此它适合成为建模中的核心单元。

`流程节点` 是流程内部的角色职责划分。  
一个节点通常对应“某个角色在这个流程里负责的一段事”。再往下拆，形成`步骤/任务`，用来表达用户动作、系统处理、数据使用和规则落点。

`流程组` 则更多是管理和阅读上的概念。  
它的作用是把数量较多的流程做主题归拢，方便维护、浏览和讨论。它可以有，但不建议把它放进正式的业务架构层级里。否则容易和阶段、子域抢概念位置。

因此，这个工具中的几个概念可以这样理解：

- `业务子域`：业务能力的划分
- `业务阶段`：业务时间线的划分
- `业务流程`：稳定的业务做法和业务单元
- `流程节点`：角色职责划分
- `步骤/任务`：具体动作与系统处理
- `流程组`：流程库中的管理分组

这里还有一个很关键的判断。  
我们认为，`流程` 不应简单地“属于某个阶段”，而应当“归属于某个子域，被阶段引用”。  
原因是，阶段表达的是时间线，而流程表达的是稳定能力。很多支撑性流程会同时服务多个阶段。如果把流程强绑定到单一阶段，后面就容易出现重复建模、阶段调整牵动流程归属、概念边界混乱等问题。

所以，更合适的理解是：

- `业务子域` 管流程归属
- `业务阶段` 管流程在时间线中的组织和引用
- `阶段视图中的节点，本质上是流程引用，而不是流程所有权`

这样处理后，产品既能表达业务全景，也能表达能力边界，两套视角可以同时成立，而不会互相打架。

所以，建模过程本身不是线性的，而是螺旋式的。  
先按时间线粗梳，再按能力边界分堆，再把阶段里的活动落到具体流程，再往下细化节点、步骤、数据和规则，最后再回过头调整前面的阶段划分、子域边界和流程颗粒度。建模不是为了把图画得完整，而是为了让团队对业务形成更稳定、更清楚、更可复用的共同认知。
