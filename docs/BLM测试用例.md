# BLM测试用例

## 1. 目标

本文档记录当前浏览器版 BLM 的核心测试范围，用于回归验证工作区编辑、确认合并、回收站恢复、未保存修改保护等关键能力。

## 2. 测试分层

### 2.1 Python 单元测试

- 覆盖后端文档迁移、工作区存储、回收站、历史快照、合并分析、接口行为。
- 命令：

```bash
python -m unittest discover -s tests -v
```

### 2.2 浏览器端到端测试

- 使用 Playwright 验证真实 UI 交互。
- 命令：

```bash
cd tools/e2e
npm run test:e2e
```

界面基线截图可通过以下脚本刷新：

```bash
cd tools/e2e
npm run capture:docs
```

## 3. 核心回归用例

### 3.1 打开文档

| 编号 | 场景 | 预期结果 |
|---|---|---|
| OPEN-01 | 点击“打开” | 弹出工作区/回收站双 TAB 弹窗 |
| OPEN-02 | 在工作区列表点击文档 | 当前文档切换成功，标题栏显示文档名 |
| OPEN-03 | 已有未保存修改时切换文档 | 先弹确认框，取消后仍停留当前文档 |

### 3.2 保存与复制

| 编号 | 场景 | 预期结果 |
|---|---|---|
| SAVE-01 | 新建文档后保存 | 工作区生成 `.json` 和 `.md` |
| SAVE-02 | 保存已存在文档 | 原文档被覆盖，旧版进入历史快照 |
| SAVE-03 | 复制文档为新名称 | 生成新文档，不覆盖原文档 |
| SAVE-04 | 复制文档使用已存在名称 | 被拒绝并提示更换名称 |

### 3.3 回收站与历史恢复

| 编号 | 场景 | 预期结果 |
|---|---|---|
| REC-01 | 删除工作区文档 | 文档进入回收站，不直接物理删除 |
| REC-02 | 打开回收站 | 可看到文档名和删除时间 |
| REC-03 | 回收站恢复 | 文档回到工作区并可重新打开 |
| REC-04 | 查看历史版本 | 可列出保存前快照 |
| REC-05 | 恢复历史版本 | 当前文档回退到目标快照，恢复前先保留当前快照 |

### 3.4 确认合并

| 编号 | 场景 | 预期结果 |
|---|---|---|
| MERGE-01 | 点击“合并” | 合并弹窗打开，左右下拉框出现两个不同文档 |
| MERGE-02 | 选择相同左右文档 | 阻止继续并提示不能选择同一文档 |
| MERGE-03 | 点击“确认合并”且无冲突 | 自动生成新的合并文档并切换到该文档 |
| MERGE-04 | 点击“确认合并”且有冲突 | 展示冲突项，不允许直接完成 |
| MERGE-05 | 完成冲突裁决后再次确认 | 合并成功，生成结果文档 |

![确认合并弹窗示意](screenshots/07_merge_dialog.png)

### 3.5 未保存修改保护

| 编号 | 场景 | 预期结果 |
|---|---|---|
| GUARD-01 | 修改后切换文档 | 弹出确认提示 |
| GUARD-02 | 修改后关闭页面 | 浏览器触发离开提醒 |
| GUARD-03 | 修改后点击删除当前文档 | 先确认，取消时保持当前文档 |

## 4. 本轮新增回归点

### 4.1 合并按钮无响应修复

- 原问题：
  浏览器版已移除路径模式，但 `app.js` 里仍残留 `S.merge.paths` 和路径回退逻辑。
- 影响：
  点击“合并”后在预填左右文档阶段抛异常，界面看起来像“没有反应”。
- 当前验证：
  - 前端结构测试确认不再依赖 `S.merge.paths`
  - Playwright 用例验证点击“合并”后可正常进入并完成确认合并

### 4.2 文档同步维护

- 检查以下文档必须存在：
  - `docs/BLM设计文档.md`
  - `docs/BLM测试用例.md`
  - `docs/BLM用户手册.md`
- 并在测试中检查核心关键词，防止代码大改后文档彻底失效。
- 打开文档、工作区编辑、确认合并等关键界面截图应通过自动截图脚本同步刷新。

## 5. 维护要求

出现以下变化时，必须同步补测试或更新用例说明：

- 打开/保存/复制/删除流程变化
- 合并规则、冲突裁决方式变化
- 回收站或历史恢复流程变化
- 工具栏按钮名称或主路径交互变化
