# BLM设计文档

## 1. 文档目的

本文档描述当前 BLM 浏览器版的核心设计，覆盖工作区文档流、合并能力、历史恢复机制、前后端模块边界，以及后续维护时需要同步更新的文档约束。

## 2. 当前产品边界

- 运行形态：浏览器版，本地 Python 服务 + 静态前端页面。
- 文档形态：工作区内一个文档对应一个 `.json`，保存时自动派生 `.md`。
- 交互模型：所有操作围绕工作区文档展开，不再支持桌面宿主与本地路径模式。
- 安全机制：覆盖保存前生成历史快照，删除时进入回收站，恢复时保留再进入工作区。

## 3. 总体架构

### 3.1 模块划分

- `blm.py`
  启动入口，负责读取环境变量并启动本地 HTTP 服务。
- `blm_core/document.py`
  负责文档标准化、旧文档兼容迁移、隐藏 UID 补齐、显示 ID 重整。
- `blm_core/storage.py`
  负责工作区文档读写、Markdown 导出、历史快照、回收站恢复。
- `blm_core/merge.py`
  负责双文档合并、冲突检测、冲突裁决应用、一致性校验。
- `blm_core/server.py`
  负责浏览器 API，暴露工作区文件、合并、恢复等接口。
- `app/`
  浏览器前端。`state.js` 管状态，`render.js/domain.js/process.js/entity.js/preview.js/manual.js` 负责界面与建模编辑，`app.js` 负责工作流编排。

![工作区编辑主界面](screenshots/06_workspace_editor.png)

### 3.2 数据流

1. 用户在浏览器打开 BLM。
2. 前端通过 `/api/files`、`/api/load/<name>` 等接口读取工作区文档。
3. 文档进入前端前会在后端统一经过迁移逻辑，保证结构稳定。
4. 用户保存时，后端先创建历史快照，再落盘 JSON 和 Markdown。
5. 用户删除时，后端将 JSON / Markdown 一并移入回收站。

## 4. 浏览器版工作区文档流

### 4.1 打开文档

- 打开入口固定为工作区文档列表。
- “打开”弹窗包含两个 TAB：
  - 工作区文档
  - 回收站
- 工作区列表支持直接打开、查看历史、删除文档。

![打开文档弹窗](screenshots/05_open_dialog.png)

### 4.2 保存与复制

- “保存”用于保存当前工作区文档。
- “复制”用于从当前文档复制出一份新文档。
- 保存时允许重命名并提示是否覆盖同名文档。
- 复制时必须使用一个不存在的新名称，不允许覆盖。

### 4.3 恢复能力

- 历史快照：
  每次覆盖保存前，旧版文档会先进入 `workspace/.history/<文档名>/`。
- 回收站：
  删除时不直接物理删除，而是移动到 `workspace/.trash/`。
- 恢复规则：
  历史恢复会先再保存当前版本快照；回收站恢复会回到原工作区名称。

## 5. 合并能力

### 5.1 当前范围

- 只支持工作区内两个文档合并。
- 不支持上传 JSON。
- 左右顺序在语义上等价。
- 默认生成一个新的合并文档。

### 5.2 前端流程

1. 用户点击“合并”。
2. 前端读取工作区文档列表。
3. 自动为左侧、右侧填充两个不同的工作区文档。
4. 点击“确认合并”后：
   - 若还未分析，则先分析；
   - 若有冲突，则要求完成裁决；
   - 若无冲突，则直接生成合并结果。

![合并文档弹窗](screenshots/07_merge_dialog.png)

### 5.3 后端规则

- 使用 `blm_core.merge` 进行双文档合并。
- 合并基于隐藏 `uid` 与名称回退规则识别对象。
- 标量冲突支持：
  - 保留左侧
  - 保留右侧
  - 自定义值
- 同名不同义对象支持：
  - 保留左侧
  - 保留右侧
  - 两者都保留
- 合并完成后统一重整显示 ID，并进行一致性校验。

![合并结果示意](screenshots/08_merge_result.png)

## 6. 文档兼容与存储策略

### 6.1 兼容策略

- 旧文档加载时通过 `migrate_document()` 自动迁移。
- 顶层 `meta` 补齐：
  - `document_uid`
  - `schema_version`
- 可合并节点补齐隐藏 `uid`。

### 6.2 存储策略

- 对外仍然保持“一个文档一个 JSON”。
- 工作区保存时同步生成 Markdown。
- 合并结果继续保存为单个 JSON，便于选择、复制、恢复和再次合并。

## 7. 关键界面设计

### 7.1 打开弹窗

- 使用工作区/回收站双 TAB。
- 列表区域独立滚动，避免文档数量多时整个弹窗失控。

### 7.2 合并弹窗

- 仅提供左右两个下拉框。
- 右下角主按钮为“确认合并”。
- 合并分析摘要、冲突项、结果预览都在同一弹窗内完成。

### 7.3 使用手册入口

- 位于顶部工具栏“导出”后方，作为独立阅读页入口，不与业务建模页签混用。
- 布局采用三栏：
  - 左侧为文档列表与目录大纲；
  - 中间为 Markdown 正文阅读区；
  - 右侧为截图速览区，点击后可跳转到正文中的对应图片。
- 文档内容来自 `docs/` 目录，截图资源来自 `docs/screenshots/`，由后端提供只读接口。
- 该页支持渲染标题、表格、列表、代码块、Mermaid 和截图，适合作为内置知识库与使用指引。

![使用手册入口与阅读页](screenshots/09_manual_tab.png)

## 8. 文档维护约定

后续代码修改时，以下文档必须同步更新：

- `docs/BLM设计文档.md`
  当架构边界、模块职责、工作流、合并规则、恢复机制发生变化时更新。
- `docs/BLM测试用例.md`
  当新增功能、修复关键缺陷、调整交互流程时补充或修改用例。
- `docs/BLM用户手册.md`
  当界面文案、操作步骤、按钮行为变化时同步更新。

当前仓库已用测试约束这些文档必须存在，并覆盖核心功能关键词，避免后续完全失联。

## 9. 截图更新方式

当前项目已提供自动截图脚本，用于在界面调整后更新文档插图：

```bash
cd tools/e2e
npm run capture:docs
```

脚本会自动启动本地 BLM、构造示例工作区文档，并更新 `docs/screenshots/` 下的文档截图。
