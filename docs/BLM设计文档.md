# BLM设计文档

## 1. 文档目标

本文档描述当前浏览器版 BLM 的核心设计，覆盖工作区文档流、合并能力、回收站与历史快照、内置使用手册，以及后续代码修改时需要同步维护的文档约束。

## 2. 当前产品边界

- 运行形态：浏览器版，本地 Python 服务加静态前端页面。
- 文档形态：工作区内一个文档对应一个 `.json`，保存时自动派生 `.md`。
- 交互模型：所有操作围绕工作区文档展开，不再支持本地上传 JSON 和桌面宿主路径模式。
- 安全机制：覆盖保存前生成历史快照，删除时进入回收站，恢复时先保护当前内容。

## 3. 总体架构

### 3.1 模块划分

- `blm.py`
  启动入口，负责读取环境变量并启动本地 HTTP 服务。
- `blm_core/document.py`
  负责文档标准化、旧文档兼容迁移、隐藏 UID 补齐、显示 ID 重整。
- `blm_core/storage.py`
  负责工作区文档读写、Markdown 导出、历史快照、回收站恢复。
- `blm_core/merge.py`
  负责双文档合并、冲突检测、冲突裁决应用、一致性校验。
- `blm_core/server.py`
  负责浏览器 API，暴露工作区文件、合并、恢复、内置文档等接口。
- `app/`
  浏览器前端。`state.js` 管状态，`render.js/domain.js/process.js/entity.js/preview.js/manual.js` 负责界面与建模编辑，`app.js` 负责工作流编排。

![工作区编辑主界面](screenshots/06_workspace_editor.png)

### 3.2 数据流

1. 用户在浏览器打开 BLM。
2. 前端通过 `/api/files`、`/api/load/<name>` 等接口读取工作区文档。
3. 文档进入前端前统一经过兼容迁移，保证结构稳定。
4. 保存时后端先生成历史快照，再落盘 JSON 和 Markdown。
5. 删除时后端把 JSON 和 Markdown 一并移入回收站。

## 4. 浏览器版工作区文档流

### 4.1 打开文档

- “打开”弹窗只面向工作区文档。
- 弹窗包含两个页签：
  - 工作区文档
  - 回收站
- 工作区列表支持直接打开、查看历史、删除文档。
- 回收站列表支持恢复误删文档。

![打开文档弹窗](screenshots/05_open_dialog.png)

### 4.2 保存与复制

- “保存”用于保存当前工作区文档。
- “复制”用于从当前文档生成一个新的工作区文档。
- 保存时允许重命名，并在同名时提示是否覆盖。
- 复制时必须使用一个不存在的新名称，不允许覆盖。

### 4.3 回收站与历史快照

- 历史快照：
  每次覆盖保存前，旧版本会先进入 `workspace/.history/<文档名>/`。
- 回收站：
  删除时不直接物理删除，而是移动到 `workspace/.trash/`。
- 恢复规则：
  历史恢复前先保护当前版本；回收站恢复后回到工作区原名称。

## 5. 合并能力

### 5.1 当前范围

- 只支持工作区内两个文档合并。
- 不支持上传 JSON。
- 左右顺序在语义上等价。
- 默认生成一个新的合并文档。

### 5.2 前端流程

1. 用户点击“合并”。
2. 前端读取工作区文档列表。
3. 左侧文档和右侧文档都通过下拉框选择。
4. 点击“确认合并”后：
   - 如尚未分析，则先分析；
   - 如存在冲突，则展示冲突项并要求裁决；
   - 如无冲突，则直接生成新的合并结果。

![合并文档弹窗](screenshots/07_merge_dialog.png)

### 5.3 后端规则

- 使用 `blm_core.merge` 进行双文档合并。
- 合并基于隐藏 `uid` 与名称回退规则识别对象。
- 标量冲突支持：
  - 保留左侧
  - 保留右侧
  - 自定义值
- 同名不同义对象支持：
  - 保留左侧
  - 保留右侧
  - 两者都保留
- 合并完成后统一重整显示 ID，并进行一致性校验。

![合并结果示意](screenshots/08_merge_result.png)

## 6. 使用手册内置阅读页

### 6.1 入口与职责

- 入口位于顶部工具栏“导出”后面，作为独立阅读入口。
- 进入使用手册后，不再显示“业务域 / 流程 / 数据 / 预览”这排建模页签。
- 使用手册页面只负责阅读 `docs/` 下的文档，不参与建模编辑。

### 6.2 当前布局

- 左侧为轻量导航区：
  - 当前用户手册目录
  - 二级目录默认折叠，避免列表过长
- 右侧为正文阅读区：
  - 顶部只保留当前文档标题
  - 不再使用高占用的说明区
  - 截图直接在正文中内联展示
- 整体目标是简洁明了，接近“左侧目录 + 右侧正文”的阅读界面。

![使用手册入口与阅读页](screenshots/09_manual_tab.png)

## 7. 文档兼容与存储策略

### 7.1 兼容策略

- 旧文档加载时通过 `migrate_document()` 自动迁移。
- 顶层 `meta` 自动补齐：
  - `document_uid`
  - `schema_version`
- 可合并节点自动补齐隐藏 `uid`。

### 7.2 存储策略

- 对外继续保持“一文档一 JSON”。
- 保存时同步生成 Markdown。
- 合并结果继续保存为单个 JSON，便于打开、复制、恢复和再次合并。

## 8. 文档维护约定

代码修改时，以下文档必须同步更新：

- `docs/BLM设计文档.md`
  当架构边界、模块职责、工作流、合并规则、恢复机制发生变化时更新。
- `docs/BLM测试用例.md`
  当新增功能、修复关键缺陷、调整交互流程时补充或修改用例。
- `docs/BLM用户手册.md`
  当界面文案、操作步骤、按钮行为变化时同步更新。

当前仓库已经用测试约束这些文档必须存在，并覆盖核心功能关键词，避免后续大改后文档失联。

## 9. 截图更新方式

项目提供自动截图脚本，用于在界面调整后刷新文档插图：

```bash
cd tools/e2e
npm run capture:docs
```

脚本会自动启动本地 BLM、构造示例工作区文档，并更新 `docs/screenshots/` 下的截图。
## 10. 2026-04 升级补充

### 10.1 流程分组模型

- 流程结构已经升级为 `业务子域 -> 流程组 -> 流程`。
- `flowGroup` 是可空字段，第一版只用于归类，不承担额外说明、模板规则或排序配置。
- 当 `flowGroup` 为空时，前端会在对应业务子域下显示为“未分组流程”。

### 10.2 建模术语升级

- 原来的 `tasks` 已经统一迁移为 `nodes`，原来的 `steps` 统一迁移为 `userSteps`。
- 节点下新增 `orchestrationTasks`，用于表达研发视角的服务编排。
- 节点主面板保持业务级可读性，编排任务改为在当前节点编辑区内直接维护，不再使用右侧二级滑出面板。
- 当前节点编辑区补充了“任务级流程图”，用于串行展示同一节点下的编排任务链路。
- 当节点切换到“任务级视图”时，抽屉顶部流程图会同步切换为“全流程任务级视图”，按流程顺序展示所有节点及各自内部的编排任务链路。
- 全流程任务级视图保留节点外框，节点内展示串行任务项，用于同时表达“流程顺序”和“节点内服务编排”两层结构。
- 用户操作步骤编辑区支持行内插入和上下移动动作，减少长步骤列表下反复滚动带来的操作负担。
- 预览和 Markdown 导出已经按“节点 -> 用户操作步骤 -> 编排任务”结构展开。

### 10.3 使用手册兼容保护

- `GET /api/runtime` 新增 `api_version` 与 `supports_docs`，用于前后端版本协商。
- 前端进入“使用手册”前会先检查运行时能力；如果本地服务过旧，会明确提示“请重启 BLM 服务”，不再误显示为“没有可展示的文档”。
- 这个保护机制主要用于处理“新前端与旧后端同时存在”的本地开发场景。

### 10.4 全景矩阵模型

- 文档新增 `panorama` 模型，用于保存全景视图的价值流列、业务域或产品边界行和单元格说明。
- 阶段新增 `panoramaColumnId`、`panoramaLaneId` 与 `panoramaSlot`，用于记录阶段在全景矩阵中的归属和单元格内行列槽位；旧文档中的 `panoramaPos` 会兼容换算为行列槽位。
- 旧文档没有 `panorama` 时会补齐一份默认模板；默认交割全景以 `交割智慧监管平台2期` 为主行，`电子仓单系统` 只作为关键环节职责参照。默认价值流为 `会员客户 / 品种参数 / 业务办理 / 风险监管`，其中入库、在库、出库统一归入“业务办理”。阶段卡片按二层大阶段建模，例如账号管理、仓库管理、质检机构管理、品种参数管理、仓单注册、仓单流转、风险监管；交割预报、仓库/厂库仓单注册等细粒度内容通过阶段内流程承载。全景阅读态增加交割语义聚合，仓库主体维护、仓库资质维护、仓房维护、垛位维护、提货地点与点位等会显示为“仓库管理”，避免单元格阶段卡片超过业务人员可扫描的数量。仍保留旧默认模板的文档会在未改过列名时自动迁移到新模板，已自定义列名的文档不强制覆盖。
- 全景编辑采用中间矩阵原地编辑，不再通过右侧抽屉维护表格。表头、行头默认按“正文 / 备注 / 标签”展示，正文单元格按“标签 / 阶段 / 备注”展示；编辑态输入框显示字段提示，未填写的元素在阅读态隐藏。新增行列的正文为空，用户主动清空正文后不会由归一化逻辑回填默认示例值。阶段卡片按行列槽位摆放，矩阵外层提供真实滚动容器和缩放能力。流程四视图的高占用说明区收敛为顶部问号提示，提示气泡通过浮层定位避免被容器裁切；“打开编辑 / 关闭编辑”等关键操作固定在顶部工具栏。
