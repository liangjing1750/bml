# Design

## 概览

- 本次重构采用“兼容迁移 + 双视角重组”的方式推进，而不是直接推翻现有模型。
- 第一原则是：流程仍然是唯一的业务资产实体；阶段中展示的是流程引用，不是流程所有权。
- 现有 `process.stageId` 与 `stage.processLinks` 会进入兼容期，通过加载时映射为新的阶段引用结构；UI 和预览逐步切换到新结构。
- 这次重构不会一口气完成所有功能，而是按“模型兼容 -> 阶段视图 -> 子域视图 -> 导出与收尾”四个切片推进。

## 核心对象

- `process`
  - 继续作为唯一流程实体。
  - 保留 `subDomain` 作为主归属字段。
  - 保留 `flowGroup` 作为流程库内管理分组。
  - 旧字段 `stageId` 进入兼容期，后续退役。
- `stage`
  - 表示业务阶段本体。
  - 负责全景图上的阶段节点信息和阶段间连线。
- `stageFlowRef`
  - 新增的阶段流程引用对象。
  - 字段建议包括：`id`、`stageId`、`processId`、`order`、`pos`。
  - 一个流程可以对应多个 `stageFlowRef`，从而进入多个阶段。
- `stageFlowLink`
  - 新增的阶段内流程引用连线对象。
  - 字段建议包括：`id`、`stageId`、`fromRefId`、`toRefId`。
- `stageLink`
  - 保留阶段之间的连线，用于业务全景图。

边界约束：

- 业务子域拥有流程归属。
- 业务阶段只拥有阶段本体和阶段关系，不拥有流程实体。
- 流程组只用于按子域浏览时的流程库管理，不进入正式业务层级。

## 流程与数据

- 旧模型基线
  - `process.stageId`：单流程单阶段归属。
  - `stage.processLinks`：阶段内部按流程 ID 连线。
  - UI 中“加入阶段”实质是修改 `process.stageId`，会把流程从一个阶段挪到另一个阶段。
- 目标模型
  - 文档顶层新增 `stageFlowRefs`。
  - 文档顶层新增 `stageFlowLinks`。
  - 阶段详情图通过 `stageFlowRefs(stageId)` 拉出节点，再通过 `stageFlowLinks(stageId)` 拉出连线。
  - 流程详情页通过 `stageFlowRefs(processId)` 反查被哪些阶段引用。
- 兼容映射
  - 如果老文档没有 `stageFlowRefs`，则从 `process.stageId` 自动生成一份默认引用。
  - 如果老文档存在 `stage.processLinks`，则基于同阶段内生成的默认引用，映射为 `stageFlowLinks`。
  - 保存阶段允许同时保留旧字段和新字段，直到 UI 全面切换完成。
- 视图组织
  - 按阶段看：
    - 业务全景图：节点是阶段，连线是 `stageLinks`。
    - 阶段详情图：节点是 `stageFlowRef`，节点内容来自对应 `process`。
  - 按子域看：
    - 左侧目录或流程库按 `subDomain -> flowGroup -> process` 组织。
    - 同一流程详情在任何视角下都只有一份。

## 关键决策

- 选择“阶段引用流程”而不是“阶段拥有流程”，是因为阶段是时间线视角，流程是稳定业务能力，二者不能长期绑定成单一上级关系。
- 阶段内节点使用引用对象而非直接使用流程实体，是为了允许同一流程多处出现且保留每处独立的布局微调能力。
- 保留 `flowGroup` 但降低其层级，是为了兼容现有流程资产管理习惯，同时防止它继续与阶段、子域争抢概念位置。
- 采用兼容迁移而不是一次性替换，是因为当前仓库已经有大量基于 `process.stageId` 的前后端逻辑和测试，直接切断风险过大。

## 风险与扩展点

- 风险 1：当前代码中有大量直接读取 `process.stageId` 的逻辑，漏改会导致阶段视图和预览不一致。
- 风险 2：旧的 `stage.processLinks` 基于流程 ID，而新的 `stageFlowLinks` 基于引用 ID，映射过程需要确保阶段内同流程多引用时不歧义。
- 风险 3：左侧目录从单树切换到双视角后，导航、返回栈和自动显位逻辑都需要同步调整。
- 风险 4：导出和预览如果同时支持新旧结构，回归范围会较大。

扩展点：

- 后续可以将 `subDomain` 从字符串升级为正式对象表，但不阻塞当前重构。
- 后续可以给阶段引用增加条件说明、展示标签和更多图布局信息。
- 后续可以把同样的“引用而非拥有”思路复用于更高层的业务地图或能力地图。
