# Requirements

## 背景

- 当前产品已经引入“业务阶段”视图，但核心数据模型仍然把流程通过 `process.stageId` 绑定到单一阶段。
- 这种做法让“阶段”同时承担了时间线组织和流程所有权两类职责，概念边界容易缠在一起。
- 对交割监管这类业务来说，很多流程属于稳定业务能力，尤其是支撑性流程，会被多个阶段共同使用。如果流程只能归属于一个阶段，就容易出现重复建模或迁移流程导致语义漂移。
- 现有左侧目录也把业务子域、业务阶段和流程组织放在一棵树里，长期会把“按能力看”和“按时间线看”这两条认知路径混在一起。

## 目标

- 明确产品中的概念边界：业务子域管理流程归属，业务阶段管理流程引用，流程组退回流程库管理分组。
- 将当前“阶段拥有流程”的实现逐步重构为“阶段引用流程”的实现，同时保持旧文档可兼容打开。
- 在界面上形成两条清晰主线：按阶段看业务全景与阶段详情，按子域看流程资产与流程组。
- 为后续大重构建立一套可持续推进的 SDD/TDD 规格与测试基线，避免多轮实现过程中口径漂移。

## 非目标

- 这一轮不引入 BPMN 级别的网关、泳道、条件表达式或复杂编排引擎。
- 这一轮不把业务子域升级成独立建模对象表，只先保证子域作为流程归属视角可持续存在。
- 这一轮不做多人在线协同、远端中心化存储或审批流平台能力。
- 这一轮不要求一次性完成所有 UI 改造，可以按兼容切片逐步落地。

## 需求

- R1. 系统应支持同一个流程被多个业务阶段引用，而不复制出多个流程实体。
- R2. 系统应保留流程的主业务子域归属，阶段只负责组织时间线，不改变流程所有权。
- R3. 阶段详情图中的节点应表示“阶段中的流程引用”，而不是流程所有权。
- R4. 用户在阶段视图中应优先执行“加入已有流程”，其次才是“新建并加入”。
- R5. 系统应支持从流程详情中回看“该流程被哪些阶段引用”。
- R6. 左侧浏览应最终支持按阶段和按子域两种组织方式，避免单一树结构承载两种认知。
- R7. 老文档中的 `process.stageId`、`stage.processLinks` 等旧结构应能在迁移期被自动映射到新结构，不丢数据。
- R8. 预览、导出、保存、合并等链路应理解新旧模型并保持结果一致。
- R9. 这次重构应通过用户旅程驱动验收，至少覆盖“加入已有流程”“同一流程多阶段复用”“双视角切换保持同一流程上下文”三条金路径。

## 验收标准

- A1. 系统存在显式的“阶段流程引用”数据结构，并能从旧文档自动生成兼容引用。
- A2. 同一流程加入两个阶段后，文档中流程实体数量不增加，但两个阶段详情图都能看到该流程。
- A3. 修改被多个阶段引用的流程名称或内容时，所有引用位置都反映同一份流程详情。
- A4. 从一个阶段移除流程引用时，不会删除流程本体，也不会影响其它阶段中的同流程引用。
- A5. 用户可在按阶段视图和按子域视图之间切换，并持续定位到同一流程。
- A6. 老文档在未手工迁移的前提下可打开、保存、预览与导出，且旧数据不丢失。
